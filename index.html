What is GitHub?

GitHub is a web platform for version control and collaboration. It lets you and others work together on projects from anywhere.

Key Facts About GitHub

  • Industry Standard: Used by millions of developers worldwide.
  • Microsoft Owned: Microsoft acquired GitHub in 2018, though it remains independently branded.
  • Free for Students: Public repositories are free, and students get premium features.
  • Career Asset: Most tech companies expect you to know Git and GitHub.

Yes, GitHub can be challenging at first. But remember: every professional developer you admire went through the same learning process. Stick with it!

Understanding Version Control

Version control is a system that records changes to files over time. Think of it as a time machine for your code.

Why Version Control Matters

  • Track Changes: See what changed, when, and by whom.
  • Undo Mistakes: Roll back to a previous version if something breaks.
  • Collaborate Safely: Multiple people can work on the same project without overwriting each other's work.
  • Backup: Your code is safely stored in the cloud.
  • Experiment: Try new features without breaking your working code.
Real World Example: Imagine writing a paper. Without version control, you might save files like "essay_final.docx", "essay_final_v2.docx", "essay_ACTUAL_final.docx". With version control, you have one file with a complete history of all changes.

Working with Repositories

A repository (or "repo") is a folder that contains your project files and the entire history of changes.

What's in a Repository?

  • Your project files (code, images, documents).
  • A README file explaining what the project is.
  • A complete history of all changes.
  • Branches for different versions or features.

The README File

The README.md file is the front page of your repository. It should explain:

  • What the project does.
  • How to use it.
  • Who created it.
Example README.md:

# My C# Learning Project
This repository contains my work for ITEC 2560.

## Author
Jane Student

## Description
Practice exercises for learning C# programming.

Branching: Experiment Safely

A branch is a separate version of your code where you can experiment without affecting the working code.

Why Use Branches?

  • Try new features without breaking working code.
  • Work on multiple features at the same time.
  • Keep your main branch clean and working.
  • Industry standard practice.
Analogy: Think of branches like save files in a video game. The main branch is your primary save. Other branches let you try different strategies without losing your progress.

Forking vs. Cloning

Students often get confused between forking and cloning. Here's the difference:

Cloning

  • Downloads a copy to your computer.
  • You can make changes locally.
  • If you have permission, you can push changes back.
  • Used for your own repositories or team projects.

Forking

  • Creates a copy of someone else's repository under YOUR GitHub account.
  • You own this copy and can do whatever you want with it.
  • Used when you want to contribute to someone else's project.
  • You can suggest changes back via "pull requests".
Real Example: Your instructor posts starter code. You fork it to your account, clone it to your computer, complete the assignment, commit and push to YOUR fork, then share the link with your instructor.

Sharing Your Repository

For Class Assignments

Option 1: Share the Link (Public Repository)

  • Make sure your repository is public.
  • Copy the URL from your browser.
  • Submit it in D2L.

Example URL: https://github.com/yourusername/your-repo-name

Option 2: Add Collaborators (Private Repository)

If you want to keep your repo private, you can give your instructor access by adding them as a collaborator in your repository settings.

For Group Projects

  • One person creates the repository.
  • They add team members as collaborators.
  • Everyone clones the same repository.
  • Use branches for different features.
  • Merge branches when features are complete.
Pro Tip: Always include a README.md that explains how to run your project. Your instructor (and future employers) will appreciate it!

Tips for GitHub Success

Yes, GitHub is Hard at First

Let's be honest: GitHub has a steep learning curve. But here's why you should push through:

It's an industry standard. Over a million developers use GitHub. It's also a career boost, since employers expect you to have a GitHub profile. You can use GitHub as a portfolio of your work. Your repos are proof of your skills. GitHub allows you to collaborate and work in teams. Finally, Microsoft, which is the second biggest software company in the world, owns GitHub and continues investing in it.

Common Mistakes to Avoid

  • Forgetting to commit before closing your computer.
  • Not pulling before you start working (gets you out of sync).
  • Committing passwords or API keys.
  • Vague commit messages like "fixed stuff".
  • Working directly on main instead of using branches.

Best Practices

  • Commit early and often.
  • Write clear commit messages.
  • Pull before you push.
  • Use .gitignore for files you don't want tracked.
  • Keep your README up to date.
  • Don't be afraid to ask for help.

Remember: Every professional developer struggled with Git at first. You're not alone. Keep practicing, and it will become second nature!
